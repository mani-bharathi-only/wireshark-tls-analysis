# TLS Encryption Traffic Analysis

## Overview

This project analyzes encrypted network communication using **Wireshark**.
The goal of this analysis is to understand how secure HTTPS connections are established using the **Transport Layer Security (TLS)** protocol.

The packet capture was generated by visiting secure websites in a browser while capturing traffic using Wireshark.

---

## Tools Used

* Wireshark
* Web Browser
* Windows Operating System

---

## Capture Details

* Capture Format: `.pcapng`
* Network Interface: WiFi
* Traffic Type: HTTPS / TLS encrypted communication

Activities performed during capture:

* Accessing secure websites
* Loading webpages
* Generating HTTPS requests

---

## Protocols Observed

During the capture several network protocols were observed:

* DNS
* TCP
* TLS
* HTTPS

These protocols work together to establish secure communication between a client and a server.

---

## TLS Handshake Analysis

### 1. Client Hello

The TLS handshake begins when the client sends a **Client Hello** message to the server.

Information included in the Client Hello packet:

* Supported TLS versions
* Supported cipher suites
* Random value
* Extensions such as Server Name Indication (SNI)

Purpose:
The client informs the server about the encryption algorithms it supports.

---

### 2. Server Hello

The server responds with a **Server Hello** packet.

Information included:

* Selected TLS version
* Selected cipher suite
* Server random value

Purpose:
The server chooses the encryption method that will be used for the session.

---

### 3. Certificate Exchange

The server sends a **digital certificate** to prove its identity.

Certificate details observed:

* Domain name
* Certificate issuer
* Public key information

Purpose:
The certificate allows the client to verify that it is communicating with the legitimate server.

---

### 4. Key Exchange

After certificate verification, the client and server perform a **key exchange process**.

This step generates a shared secret key used for encrypting the session.

Common key exchange mechanisms include:

* ECDHE
* RSA

---

### 5. Encrypted Communication

Once the handshake is completed, the communication becomes encrypted.

In Wireshark this traffic appears as:

TLS Application Data

The actual contents of the packets cannot be read because they are encrypted.

Purpose:
This encryption ensures:

* Confidentiality
* Data integrity
* Secure communication over the internet

---

## Observations

* TCP three-way handshake occurs before the TLS handshake.
* TLS handshake establishes encryption between the client and server.
* After the handshake, packets are encrypted and appear as TLS Application Data.
* HTTPS communication relies on TLS to protect user data.

---

## Conclusion

The analysis demonstrates how TLS establishes a secure communication channel between a client and a server.

By observing the TLS handshake in Wireshark, it is possible to understand how encryption is negotiated and how secure HTTPS communication begins.

Although the encrypted data cannot be read directly, the handshake packets provide valuable insight into the security mechanisms used in modern web communication.
